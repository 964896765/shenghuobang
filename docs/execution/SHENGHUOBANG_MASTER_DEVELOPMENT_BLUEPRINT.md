# 生活帮产品最终形态与系统开发总蓝图

## 1. 最终定位

**生活帮——连接创意、生产、使用、流转和回收的产品全生命周期协作与资源交易平台。**

普通用户理解为：买卖闲置、发布需求、找人解决问题，也可以把一个想法真正做出来。

企业和产业链理解为：连接设计、工程、生产、销售、维修、回收和材料再利用的一站式协作平台。

主品牌表达：

- 产品副标题：让创意落地，让物品持续流转；
- 普通用户宣传：有需求，找生活帮；有闲置，上生活帮；有想法，也能在生活帮做出来；
- 核心链路：想出来 → 做出来 → 买到手 → 用起来 → 修得好 → 转出去 → 回收掉 → 再利用。

## 2. 三层产品结构

### 第一层：生活服务与资源流转

面向普通用户，承担日常使用和用户增长：需求、工程服务、旧物、出售、购买、置换、赠送、维修、租赁、回收、同城服务、订单和售后。

### 第二层：创意落地与新品孵化

面向创意者、设计师、工程师和创业团队：创意、保密、协作者、项目、设计版本、原型、测试、意向登记、预售筹措、生产对接和新品发布。

### 第三层：产品生命周期与产业链追踪

面向组织和企业：产品型号、设计发布、BOM、RFQ、生产订单、批次、质检、产品实例、所有权、维修、翻新、回收、拆解、零件和再生材料。

三层不是三个独立 App。它们共享统一账号、身份、组织、项目、物品、订单、资金、文件、消息、权限和审计。

## 3. 最终系统骨架

### 六个参与者概念

1. Account：登录账号；
2. Business Identity：一个账号可有多个业务身份；
3. Qualification：身份或组织的认证资质；
4. Organization：工作室、门店、公司、工厂、机构；
5. Project Role：某个项目内的角色；
6. Platform Staff Assignment：客服、审核、财务和安全等后台职务。

### 权限判定

使用：RBAC + ABAC + Data Scope + Field Mask。

服务端判定：

```text
能力是否存在
+ 账号是否有效
+ 身份/认证是否满足
+ 是否为有效组织成员
+ 是否为项目成员/被分配执行人
+ 是否拥有该资源或被授权
+ 当前状态是否允许
+ 保密级别/NDA 是否允许
+ 是否违反职责分离
= allow / deny + reasonCode + dataScope + fieldMask
```

客户端工作台切换不是权限来源。

## 4. 最终 App 形态

固定五 Tab：主页、发现、发布、消息、我的。

普通用户首页只展示：发布需求、附近服务、附近旧物、回收估价、维修、置换赠送、创意专区和进行中事项。

专业用户从“我的 → 工作台切换”进入设计师、工程师、商家、维修、生产商、供应商、质检、回收商、企业或平台运营工作台。

普通首页不得展示 BOM、RFQ、生产批次、材料追踪、财务审核和安全审计。

详细页面结构见 [最终 App 信息架构](../strategy/END_STATE_APP_INFORMATION_ARCHITECTURE.md)。

## 5. 三个核心闭环

### 交易闭环

发现 → 沟通 → 报价 → 下单 → 托管 → 交付 → 验收 → 结算 → 售后。

### 产品闭环

创意 → 设计 → 原型 → 生产 → 销售 → 使用 → 维修 → 易手 → 翻新 → 回收。

### 材料闭环

材料 → 零部件 → 产品 → 使用 → 拆解 → 再生材料 → 新产品。

## 6. 主版本顺序

### V3.3-A：身份、组织与数据权限基础工程

**完成后用户能看到：**

- 普通用户原体验保持；
- 我的身份和认证；
- 我的组织和成员；
- 工作台切换；
- 多成员项目；
- 最小专业工作台。

**完成后系统具备：**

- 多身份和认证分离；
- 组织数据隔离；
- 项目成员和项目角色；
- 能力权限、数据范围、字段脱敏；
- 职责分离；
- 停职、退出和权限回收；
- 旧模型兼容迁移。

**不做：** 完整创意、生产、回收产业链页面。

### V3.3-B：创意落地最小闭环

**完整流程：**

发布创意 → 公开/私密/NDA → 邀请设计师/工程师 → 接受邀请 → 创意转项目 → 设计版本 → 原型里程碑 → 成果提交 → 验收/返工 → 意向登记。

**核心参与者：** 发起人、设计师、工程师、项目负责人、受邀查看者。

**不做：** 完整 RFQ、BOM、量产和真实预售资金。

### V4.1：生产协作

**完整流程：**

设计冻结 → RFQ → 组织邀请 → 授权资料 → 独立报价 → 选定合作方 → 设计发布 → BOM → 生产订单 → 试产 → 检测 → 工程变更 → 量产 → 批次质检 → 产品实例。

**关键隔离：** 竞争报价隔离、图纸授权、生产商不能覆盖设计版本、质检机构只看指派批次。

### V4.2：产品数字档案与流转

**完整流程：**

产品型号 → 单件实例 → 首次销售 → 所有权 → 使用 → 维修/部件更换 → 成色/折损/估值 → 易手/赠送/置换/租赁 → 翻新 → 回收建议。

新品和旧物连接到同一物品事实链。历史所有者个人信息不随所有权转移。

### V4.3：回收与材料闭环

**完整流程：**

产品实例 → 回收申请 → 初估 → 回收商报价 → 上门/邮寄 → 复检 → 最终价格 → 交接 → 结算 → 拆解 → 零件分流 → 材料重量/纯度 → 再生材料批次 → 新产品关联。

## 7. 现有代码如何复用

直接复用并扩展：

- needs 作为生活需求和协作需求基础；
- quotes 作为服务/设计/工程报价基础；
- projects/milestones/requirements/changes/files/acceptances 作为创意项目基础；
- items/listings/orders 作为产品流转基础；
- ownership/service/status history 作为生命周期基础；
- recycling requests/quotes 作为回收入口基础；
- conversations/messages/notifications 作为全平台协作基础；
- payments/refunds/escrow/settlements 作为统一资金基础；
- complaints/audit logs 作为争议和治理基础。

不推倒重写；通过追加表、适配器、双读/回填和明确退役门禁逐步升级。

详细关系见 [核心领域整合蓝图](../architecture/CORE_DOMAIN_INTEGRATION_BLUEPRINT.md)。

## 8. 跨阶段技术轨

### 金额与财务

整数分、历史回填、双写、读切换、对账。任何阶段款、生产采购和回收结算上线前必须完成对应门禁。

### 云环境

隔离 MySQL、HTTPS/WSS、对象存储、Push、日志、指标、备份和恢复。外部协作前必须有稳定测试环境。

### 支付

先沙箱后正式；回调验签、幂等、金额匹配、托管、退款、结算和职责分离。

### 安全与隐私

默认拒绝、字段脱敏、文件撤权、访问目的、密钥、注销、数据导出和审计。

### AI、位置和搜索

AI 用于整理需求、识别物品和估价辅助，不可决定权限、付款、认证或质检；位置按需授权并最小精度；搜索结果先做权限过滤。

### 发布质量

每阶段必须包含迁移、回归、越权、并发、真机、弱网和恢复验证。

## 9. 统一开发方法

每个主阶段固定七批：

1. 规格和威胁模型；
2. 追加迁移和回填；
3. 服务端领域服务、权限、事务和审计；
4. 客户端入口、工作台和流程；
5. 自动化、越权和并发测试；
6. MySQL、真机、弱网和外部沙箱；
7. 报告、源码归档和下一阶段门禁。

Codex 每次只执行一个批次，完成后停止。逐批清单见 [主阶段交付总矩阵](./MASTER_STAGE_DELIVERY_MATRIX.md)。

## 11. 最终完成判断

生活帮最终不是“页面很多”就算完成，而要同时满足：

- 普通用户可以简单完成需求、旧物、维修和回收；
- 创意可以变成项目和原型；
- 项目可以进入生产并形成产品实例；
- 产品实例可以维修、易手、翻新和回收；
- 回收产出可以形成零件或再生材料并回到新产品；
- 每一步都有明确参与者、权限、文件、状态、订单、资金和审计；
- 不同组织、项目和历史所有者的数据不会越权泄露；
- 所有关键事实可以从结果反向追溯到来源。
