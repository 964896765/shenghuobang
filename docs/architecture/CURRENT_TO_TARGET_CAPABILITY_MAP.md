# 当前能力到最终平台的复用与缺口地图

## 1. 结论

当前项目已经具备普通用户交易与工程项目的可复用骨架，不应重写。后续采用“适配、抽象、扩展”方式：先解除当前一对一角色限制，再把已有项目和物品能力接入新的身份、组织和生命周期模型。

## 2. 现有能力复用

| 现有能力 | 当前实现 | 最终用途 | 处理方式 |
|---|---|---|---|
| 需求 | `needs`、评论、支持、AI 结构化 | 生活需求、设计需求、创意协作招募 | 保留；创意使用独立主档并可生成需求 |
| 报价 | `quotes`、`quote_versions` | 服务报价、设计/工程报价 | 保留；增加报价主体 identity/org 与可见范围 |
| 项目 | `projects` | 创意落地、维修、定制和后续生产项目 | 保留；由双人项目升级为多成员项目 |
| 里程碑 | `milestones` | 设计、原型、测试、生产节点 | 保留；增加负责人、验收人和依赖 |
| 文件交付 | `project_files`、短签名和访问日志 | 图纸、方案、原型、质检和协议 | 保留；增加保密级别、授权范围、设计版本元数据 |
| 项目需求与变更 | `project_requirements`、`project_changes` | 正式需求、工程变更和范围控制 | 保留并扩展批准矩阵 |
| 验收 | `project_acceptances` | 里程碑交付验收 | 保留；增加职责分离与多人验收策略 |
| 认证 | 实名/工程师/商家三套表 | 通用个人与组织资质 | 迁移至统一 qualification 模型 |
| 旧物发布 | `items`、`listings`、多种流转模式 | 产品流转层 | 保留并接入产品型号和数字档案 |
| 回收 | `recycling_requests/quotes` | 回收入口 | 保留前端入口；后续扩展复检、拆解和材料去向 |
| 所有权和维修 | ownership/service/status history | 产品生命周期 | 直接作为 V4.2 基础 |
| 订单 | `orders`、状态日志 | 所有需要履约和资金的业务 | 保留统一订单，不为新模块另造订单 |
| 支付/退款/托管/结算 | 已有账务表和沙箱流程 | 正式资金闭环 | 完成金额、支付和对账技术轨后复用 |
| 消息和通知 | conversation/message/notification/push | 全部协作场景 | 保留并增加项目/组织频道和权限 |
| 投诉和审计 | complaint + audit | 交易、项目和平台治理 | 保留；增加职责分离和资源快照 |
| 位置 | 城市和约 1km 精度偏好 | 附近服务、旧物和回收 | 保留按需授权原则 |

## 3. 当前关键限制

- 账号通过 `currentRole` 在 user/engineer/merchant 三选一切换，不能表达多身份并存。
- `engineerStatus`、`merchantStatus` 把身份和认证揉在一起。
- 商家、维修商和回收商主要被压缩在 merchant 档案中，没有组织和员工模型。
- 项目仅通过 `ownerId + engineerId` 表达双人关系。
- 路由大量使用资源 ID 直接比较，没有统一授权决策和字段脱敏。
- 后台职务存放在 `users.role`，无法表达有效期、职责分离和案件分配。
- 项目文件已有访问保护，但缺少 NDA、保密级别和组织/项目授权策略。

## 4. 不重写原则

- 不删除或一次性替换现有项目、物品、订单和资金表。
- 不先建立几十种行业专属表和页面。
- V3.3-A 只做追加表、适配器和授权服务。
- V3.3-B 使用现有项目/里程碑/文件能力验证新底座。
- 每个旧字段只有在新路径双读、回填、验证和回滚均通过后才停止使用。
