# V3.2.4 当前验收基线

## 1. 文档定位

- 本文档定义 V3.2.4 当前版本的验收边界、必审范围与判定标准。
- 产品蓝图定义长期最终方向；本文档只定义 V3.2.4 当前版本范围。

## 3. 当前版本目标

- V3.2.4 的重点是：
  - 全程序审查。
  - Android 真机验收。
  - 问题冻结与优先级归类。
  - 体验收口。
- V3.2.4 不是新增整套业务的大版本。

## 4. 环境与证据前提

- 局域网地址只允许存在于被忽略的 `.env` 与 `.env.local` 中。
- 从被忽略的 `.env` 和 `.env.local` 读取当前局域网地址，并把本次实际地址记录在仓库外测试证据中。
- 真机证据与自动化证据分开记录。
- 未经人工或自动化证据确认的能力，不得写成“已通过”。

## 5. 工程通过线

- 至少通过以下命令：
  - `pnpm check`
  - `pnpm lint`
  - `pnpm test`
  - `pnpm build`
  - `pnpm build:web`
- 涉及数据库、支付、投诉、物品、消息、文件或 WebSocket 的功能，还必须通过：
  - `docker compose up -d`
  - `pnpm db:migrate`
  - `pnpm db:seed`
  - `pnpm test:integration:mysql`
  - `pnpm test:integration:v32`
  - `pnpm test:migrations:empty`
- 运行时至少检查：
  - `/api/health`
  - `/api/ready`
  - Metro LAN 可通过当前 IPv4 访问

## 8. V3.2.4 只检查现状项

- 投诉后台。
- 财务后台。
- 审计后台。
- 后台认证审核中心。
- 后台菜单入口。

以上项目只执行页面存在性、后端权限、接口契约、审计记录和已有自动化证据审查，不因后端 Procedure 存在而新增 UI，也不执行人工业务写操作。

## 10. 不纳入缺陷统计的边界

- `FUTURE` 与当前版本“不要求完成”的能力，不计入 V3.2.4 缺陷。
- 开发环境使用沙箱支付、开发级 Push、本地存储或 Stub 时，只要文案和行为边界清楚，不能直接判定为缺陷。
- 局域网地址漂移优先按环境问题记录；仅当产品反馈不清楚时，才登记为体验问题。

## 11. 验收结论判定

- 通过：
  - 工程通过线满足。
  - 必须通过项均有证据。
  - 无未解决 P0。
- 有条件通过：
  - 工程通过线满足。
  - 无 P0。
  - 仍存在少量不阻断主链路的 P1。
- 不通过：
  - 存在阻断性崩溃、无法联调、无法登录、必须通过项中断、状态错误或数据错误。

## 12. 审查输出要求

- 每项审查都必须记录：
  - 检查编号。
  - 参照文件和章节。
  - 页面或接口。
  - 用户角色。
  - 前置数据。
  - 操作步骤。
  - 预期结果。
  - 只读或写操作。
  - 自动化或人工。
  - 证据文件。
  - 当前状态。
  - 问题编号。
- 审查阶段只登记差距，不自动发起修复。
