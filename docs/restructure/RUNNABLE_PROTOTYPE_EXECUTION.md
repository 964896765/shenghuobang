# 生活帮高可信产品全生命周期平台：可运行雏形执行文件

## 1. 文件用途

本文件是 `codex/v4-product-restructure` 分支的**唯一持续执行记录**。本轮工作的目标不是把现有系统推倒重写，也不是一次性堆满全部目标页面，而是在 V3.2.4 稳定业务能力与已经落地的 V3.3-A、V3.3-B 能力上，逐步形成可以在 Android 与 Web 上真实运行、可以连接 Express、tRPC 与 MySQL、可以用固定演示数据复现的产品全生命周期雏形。

每个里程碑只允许有一个明确交付主题。里程碑完成后必须更新本文件中的状态、自动化结果、人工验证结果、未完成项与风险，再进入下一里程碑。任何页面如果仍是静态数据、占位按钮或临时跳转，都不能标记为完成。

> **产品主线：**生活问题与需求产生创意，创意进入协作与新品筹措，成熟方案形成产品型号与单件身份，产品通过商城和订单流转，在使用中沉淀服务、评价、信用与内容，最终进入维修、转赠、二手或回收，并通过可追溯事件形成可信生命周期。

## 2. 基线与决策优先级

当前源码继续使用 Expo 54、React Native 0.81、Expo Router、TypeScript、Express、tRPC、MySQL 8、Drizzle ORM、TanStack Query、NativeWind 与 pnpm，不更换为其他技术栈。[1]

仓库的稳定业务基线仍为 V3.2.4，但当前分支已经包含 V3.3-A 的身份、组织、权限与工作台能力，以及 V3.3-B 的创意、协作者、设计原型和验收意向能力。`README.md`、`todo.md` 中“当前唯一任务为 A1”的文字已落后于实际提交和验证脚本；本文件记录的是用户明确批准后的产品重构执行主线，不把旧总控文字误当成当前代码事实。[2] [3]

如本文件与历史 UI 草案冲突，以最终产品形态、最终信息架构、核心领域整合蓝图、身份组织权限目标架构及本轮用户确认的总体规划为准。[4] [5] [6] [7]

| 决策层级 | 依据 | 用途 |
|---|---|---|
| 1 | 用户当前明确任务与总体规划 | 决定本轮产品目标和允许推进的范围 |
| 2 | 本文件 | 控制当前里程碑、验收与连续执行 |
| 3 | 最终产品形态、信息架构、核心领域与权限架构 | 约束长期结构和领域边界 |
| 4 | 当前代码、迁移和自动化测试 | 判断真实已有能力与兼容边界 |
| 5 | README、todo、历史变更说明和历史 UI 草案 | 提供运行方式与历史背景，不覆盖新决策 |

## 3. 不可破坏的工程规则

所有业务页面必须通过真实 tRPC Procedure 连接服务端，再由服务端读写 MySQL。客户端工作台与按钮显示只负责体验，不是授权边界；服务端继续使用 RBAC、ABAC、Data Scope 与 Field Mask 的组合授权。账号、业务身份、认证、组织岗位、项目角色和平台职务不得混为一个 `users.role` 枚举。[6]

现有订单、资金、投诉、认证、文件、消息和审计底座必须复用。业务模块只能创建支付或资金意图，不能直接把订单或支付状态改成成功。当前真实渠道尚未接入，演示必须明确使用 `SandboxPaymentProvider`，不得声称发生真实资金交易。[1] [8] [9]

旧迁移不得删除或改写。任何 Schema 变化必须使用新的只追加 Drizzle 迁移，并验证空库迁移和至少一个适用的旧版本升级路径。历史整元 `INT` 与部分 `DECIMAL` 账本仍处于金额迁移专项中；除非进入明确的影子列阶段，否则本轮不得引入新的小数业务金额语义。[8] [9]

所有关键事实必须保留历史，禁止静默覆盖。产品、物品、所有权、维修、质检、订单、退款、回收、认证、成员和高风险授权的变化都必须保留状态日志、事件、版本或审计记录。[5]

## 4. 当前能力盘点与复用映射

### 4.1 核心领域复用清单

| 目标领域 | 当前真实底座 | 本轮复用方式 | 首要增量 |
|---|---|---|---|
| 账号、身份与组织 | `users`、身份与 profile、认证、组织、成员、项目成员、工作台偏好、平台职务 | 保持 Account 与 Business Identity 分离；所有新增专业记录绑定身份、组织或项目数据范围 | 仅补目标领域需要的关系，不新增平行用户体系 |
| 需求与问题 | `needs`、评论、方案、报价、需求 AI 整理 | 继续作为普通用户“生活问题”的低门槛入口 | 增加来源追踪和“转为创意/项目”的显式关系 |
| 创意协作 | ideas、可见范围、NDA、邀请、协作者、转项目、设计版本、原型、验收意向 | 直接复用 V3.3-B 已有服务、路由和页面 | 补新品筹措与产品型号的下游连接 |
| 项目协作 | `projects`、成员、需求版本、里程碑、文件、变更、验收、投诉 | 项目继续管理范围、成员、版本、任务和验收 | 关联筹措、设计发布与产品型号，保持项目和订单状态机独立 |

| 资金与账本 | payments、attempts、events、refunds、escrow、settlements | 继续使用 Provider Registry、幂等键、事务和不可删除事件 | 新品订单和筹措意向接入沙箱资金意图；不伪造生产支付 |
| 评价与信用 | reviews、credit events、信用中心 | 保留现有订单评价和信用事件 | 增加来源标签、规则版本、复核/举报状态及内容侧可信摘要 |
| 文件与保密 | `stored_files`、访问日志、项目文件、短时签名 | 设计、证明、质检和回收文件继续进入统一存储 | 产品护照按公开/本人/内部生成脱敏视图 |
| 消息与通知 | conversations、messages、WebSocket、notifications、Push Provider | 新流程继续复用统一会话和通知路由 | 补新品筹措、产品服务、维修、回收关键事件通知 |
| 投诉与审计 | complaints、活动锁、裁定、资金动作、audit logs | 高风险动作保持职责分离、二次确认和 reasonCode | 新增领域的状态转换与敏感查询进入审计 |

### 4.2 当前可直接演示的闭环

当前代码已经能够演示需求发布与报价、报价转项目、里程碑与验收、物品发布、二手购买或出价、赠送申请、置换、回收询价与报价、订单、沙箱支付、退款、评价、信用中心、消息、通知、文件访问与投诉。以上能力应保留并作为后续端到端演示的底座，而不是在重构中重写。[1] [10]

### 4.3 当前关键缺口

| 缺口 | 当前证据 | 雏形完成标准 |
|---|---|---|
| 首页与发现未按生命周期主线组织 | 首页主要聚合需求、工程师与旧物；发现仍偏历史分类 | 首页能进入需求、创意、商城与可信产品；发现能统一浏览需求、创意、产品和内容 |
| 缺少统一产品型号与单件身份 | 当前以 item/listing 为主，没有独立 ProductModel/ProductUnit 路由 | 一件演示产品可从型号进入单件身份，并显示设计、制造、所有权、服务和回收事件 |
| 缺少新品筹措最小闭环 | 已有 idea/project/intention，但没有 Campaign/Pledge 主链 | 发起人可创建新品筹措活动，普通用户可提交真实意向，活动统计来自数据库 |
| 缺少新品商城骨架 | 已有 listing/order/payment，但没有 Product/SKU/Cart | 演示 SKU 可加入购物车、创建统一订单并进入沙箱支付 |
| 维修只存在历史记录，缺独立工单 | `item_service_history` 可记录事实，但没有 RepairOrder 主流程 | 用户可创建维修工单，服务方处理，完成后回写服务历史 |
| 回收停留在询价到订单 | 没有复检、交接、拆解和材料批次 | 至少完成回收交接与证明事件；拆解和材料批次可作为后续里程碑 |
| 信用与内容尚未统一 | 订单评价与信用存在，内容发现不足 | 评价、信用事件与内容具有来源标签并可从发现入口查看 |

## 5. 信息架构与导航目标

普通用户默认保持五个底部入口：**首页、发现、发布、消息、我的**。专业能力通过“身份与工作台”进入，不为每种身份复制一套底部导航。首页顶部位置入口必须是全局唯一位置入口；发现页不得重复放置第二个城市入口。[4]

发布中心固定为业务类型选择器，首批至少包含“需求、创意、物品、内容”四类真实入口；后续增加服务与回收快捷入口时仍应进入各自正式表单。发布中心不得成为装饰性按钮集合。

### 6.2 第一批雏形最低范围

第一批必须至少打通四条链路：

1. **需求—创意—新品筹措**：普通用户发布生活问题或创意，发起人创建新品筹措活动，其他用户提交真实数据库意向。
2. **产品—商城—订单—沙箱支付**：一个 ProductModel 对应至少一个 ProductUnit 与一个可售 SKU，用户加入购物车、创建统一订单并进入 Sandbox 支付。
3. **使用—评价—信用—内容**：订单完成后可以评价，信用事件可查看，发现页能够读取具有真实来源的内容或可信摘要。
4. **维修—捐赠—二手—回收**：同一件产品可进入维修、赠送、二手或回收路径，关键状态进入产品护照时间线。

以上链路可以在同一演示产品上串联，不要求首批覆盖所有高级产业链角色，但必须真实读写数据库并保留状态历史。

## 7. 数据与 API 设计约束

新增表优先使用以下统一归属字段：`ownerAccountId`、`subjectIdentityId`、`organizationId`、`projectId`、`productModelId`、`productUnitId`、`orderId`、`sourceResourceType` 与 `sourceResourceId`。服务端必须先解析数据范围，再执行数据库过滤；禁止读取全量数据后只在前端隐藏。[5] [6]

| 要求 | 实施方式 |
|---|---|
| 状态机 | 通过领域服务校验允许的前态和目标态，禁止路由层直接任意更新最终状态 |
| 幂等 | 创建订单、支付、筹措意向、产品单件、所有权转移等使用业务幂等键和数据库唯一约束 |
| 历史 | 状态、版本、所有权、服务、护照和资金变化写独立事件或日志 |
| 审计 | 敏感允许/拒绝、权限变化、内部护照、质检、资金和回收证明访问写审计 |
| 脱敏 | 护照、认证、供应链和交易信息按公开、本人、成员、组织内部和平台职务返回不同视图 |
| 并发 | 库存扣减、接受报价、选择领取人、所有权转移和回收交接使用事务、锁或唯一约束 |

## 8. 演示数据策略

演示数据继续采用可重复 upsert，不清空数据库、不删除其他用户数据。固定账号继续沿用仓库现有六个演示账号，新增产品、SKU、筹措、购物车、订单、维修与护照数据必须使用稳定演示标识，并通过幂等测试证明连续执行不会重复。[1]

首个完整演示建议围绕一款“可维修、可转赠、可回收的小型家用产品”建立单一主线。演示数据至少包含一个需求、一个创意、一个项目、一个筹措活动、一个产品型号、两个单件身份、一个 SKU、一个订单、一笔沙箱支付、一条评价、一条维修信息和一条回收证明事件。

## 9. 测试与验收门禁

每个里程碑至少运行 `pnpm check`、`pnpm lint`、`pnpm test`、`pnpm build` 与 `pnpm build:web`。涉及数据库、订单、资金、投诉、物品、文件、消息或 WebSocket 时，还必须运行相应 MySQL 集成脚本。[1]

最终候选至少执行以下命令，并把结果写回本文件：

```bash
pnpm check
pnpm lint
pnpm test
pnpm build
pnpm build:web
pnpm test:integration:mysql
pnpm test:integration:v32
pnpm test:migrations:empty
pnpm test:location:mysql
pnpm test:seed:idempotent
```

V3.3-A 与 V3.3-B 已有专项测试不得删除或跳过；新领域应沿用相同的服务、路由、App 合约和真实 MySQL 分层验证方式。Android 首先验证可生成原生包与 Metro bundle，再在可用设备或模拟器上检查登录、首页、发布、消息、订单、支付提示、返回键、键盘、图片权限与生命周期主链。缺少真实 EAS、FCM/APNs、支付或第三方核验凭据时，必须明确写为未验证，不得用 Stub 结果冒充生产验证。

## 10. 当前风险与处理原则

| 风险 | 影响 | 处理原则 |
|---|---|---|
| 旧文档与分支实际能力不一致 | 可能误退回 A1 或重复开发 | 以 Git 提交、Schema、路由和自动化脚本为事实源；在交付阶段同步 README/todo |
| 目标范围很大 | 容易出现大量半成品 | 严格按 M0—M9，一次只完成一个可验证里程碑 |
| 产品型号、物品、商品和 listing 混淆 | 历史与交易语义错误 | ProductModel 表示型号，ProductUnit 表示单件，Product/SKU 表示售卖，listing 表示一次流转 |
| 订单与项目混淆 | 协作和履约状态互相污染 | Project 与 Order 保持独立状态机，通过来源和业务类型关联 |
| 金额单位迁移未完成 | 新旧业务金额不一致 | 本轮不引入小数业务金额；所有资金继续走统一后端校验和账本 |
| 页面先行导致假闭环 | UI 看似完成但无数据事实 | Schema、服务、授权、路由和测试先于页面；每个按钮必须调用真实 Procedure |
| 外部服务缺少凭据 | 无法证明生产能力 | 使用已声明的 Provider/Stub，并在 UI 和文档中明确边界 |
