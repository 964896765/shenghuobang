# 产品路线图

## APK 构建规则

- 纯 JS/TS、页面、样式、后端和文档修改：通常不需要重新构建 APK。
- 原生依赖、Expo Plugin、Android 权限、原生配置或 SDK 变化：需要重新 EAS Build。
- 是否重建 APK 必须按每个阶段的实际改动判断，不能默认所有修复都要求重建。

## V3.2.4 全量审查与体验收口

- 阶段目标：冻结产品标准、完成全程序审查、Android 真机验收、冻结问题清单。
- 进入条件：产品蓝图、基线、矩阵、清单、追踪表和状态机文档齐备。
- 必须完成：全量审查、证据归档、问题分级、体验收口标准冻结。
- 明确不做：新增大业务、正式支付、云存储、正式 Push、iOS 发布。
- 数据库影响：不允许生产变更；审查期仅允许受控测试环境写操作。
- 后端影响：不扩展新业务接口，重点核对现有状态机和权限。
- 移动端影响：重点为 Android Development Build 真机验收。
- 是否需要重新构建 APK：通常不需要；仅当后续修复涉及原生依赖、Expo Plugin、Android 权限、原生配置或 SDK 变化时，才需要重新 EAS Build。
- 自动验收：基础命令、迁移、Seed、集成测试、运行时测试。
- 真机验收：Android 首页到消息、发布、订单等主路径。
- 退出条件：问题清单冻结，具备进入修复批次条件。

## V3.2.5 发布候选与产品合规

- 阶段目标：处理 V3.2.4 冻结问题，收敛发布候选范围，补齐关键合规说明。
- 进入条件：V3.2.4 审查已完成，问题清单已冻结。
- 必须完成：P0 清零、核心 P1 收敛、发布候选说明、权限与隐私文档更新。
- 明确不做：云端生产化、金额迁移、正式支付。
- 数据库影响：允许小规模补丁迁移。
- 后端影响：修复缺陷、完善权限与状态约束。
- 移动端影响：需要重新构建 Android Development Build 或候选包。
- 是否需要重新构建 APK：按实际改动判断；纯 JS/TS、页面、样式、后端和文档修复通常不需要，原生相关变更需要。
- 自动验收：全部基线命令 + 相关回归测试。
- 真机验收：重点回归 P0/P1 相关路径。
- 退出条件：形成可发布候选版本。

## V3.2.6 云端测试环境

- 阶段目标：建立可重复部署的云端测试环境。
- 进入条件：发布候选版本稳定。
- 必须完成：云端测试部署、环境隔离、基础监控、测试域名。
- 明确不做：正式生产支付与正式 Push 全量上线。
- 数据库影响：新增云端测试库与迁移流程。
- 后端影响：环境变量、部署脚本、健康检查强化。
- 移动端影响：切换测试环境配置与验收路径。
- 是否需要重新构建 APK：需要，因为环境配置和原生连接目标通常会随测试环境切换一起重新验证。
- 自动验收：云端 `/api/health` `/api/ready`、部署脚本验证。
- 真机验收：云端环境登录、浏览、消息、上传基础链路。
- 退出条件：云端测试环境稳定可用。

## V3.2.7 云存储与正式 Push

- 阶段目标：替换本地文件存储与开发级 Push 为正式环境能力。
- 进入条件：云端测试环境稳定。
- 必须完成：对象存储、正式 Push Provider、Token 生命周期和失败重试验证。
- 明确不做：金额迁移与正式支付。
- 数据库影响：文件元数据与投递记录可能扩展。
- 后端影响：存储适配器、Push Provider、重试机制增强。
- 移动端影响：通知权限、Push 收发、文件上传下载需回归。
- 是否需要重新构建 APK：需要，因为通知与原生能力、权限和凭据通常要重新验证。
- 自动验收：Push 与文件相关自动化增强。
- 真机验收：通知到达、点击跳转、图片与文件访问。
- 退出条件：云存储与正式 Push 达到测试环境稳定。

## V3.3.0 金额迁移

- 阶段目标：完成金额体系迁移与历史账本对齐。
- 进入条件：云端基础能力稳定。
- 必须完成：金额模型方案、迁移脚本、历史账本校验、回归测试。
- 明确不做：正式支付全量接入。
- 数据库影响：高，涉及订单、支付、退款、结算等关键表。
- 后端影响：金额校验、服务层和审计逻辑更新。
- 移动端影响：金额展示、输入、校验与文案回归。
- 是否需要重新构建 APK：按实际改动判断；如仅为后端与金额规则调整通常不需要，如涉及原生依赖或 SDK 变更则需要。
- 自动验收：金额专项迁移与对账测试。
- 真机验收：涉及金额的主路径回归。
- 退出条件：金额体系与历史数据完成一致性验证。

## V3.3.1 正式支付

- 阶段目标：接入正式支付与生产回调。
- 进入条件：金额迁移完成。
- 必须完成：正式支付 Provider、回调、对账、退款链路。
- 明确不做：新增大业务线。
- 数据库影响：支付与退款事件模型可能扩展。
- 后端影响：支付服务、签名校验、回调处理、对账任务。
- 移动端影响：支付页、支付结果页、错误态与重试流程回归。
- 是否需要重新构建 APK：通常需要，因为支付相关原生依赖、权限或 SDK 接入会触发重新构建。
- 自动验收：支付回调与对账脚本。
- 真机验收：真实支付或正式沙箱路径。
- 退出条件：正式支付链路可控且通过合规验证。

## V3.4 / V4.0 赠送、捐赠、租借、拍卖、新品共创与运营后台

- 阶段目标：扩展长期产品能力与运营工具。
- 进入条件：基础交易、支付、云能力稳定。
- 必须完成：逐个业务线 PRD、状态机、权限、风控与运营后台设计。
- 明确不做：在没有产品确认前一次性并入全部能力。
- 数据库影响：高，涉及新增实体与状态机。
- 后端影响：新增领域服务、状态机与后台权限。
- 移动端影响：新增入口、详情、创建页、运营页。
- 是否需要重新构建 APK：按每条新业务线实际改动判断；涉及新增原生依赖或权限时需要重新构建。
- 自动验收：按每条新业务线单独建立。
- 真机验收：按每条新业务线单独建立。
- 退出条件：每个新增业务线拥有独立验收基线并完成冻结。
