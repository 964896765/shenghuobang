# 生活帮产品蓝图

## 1. 文档定位

- 本文档是“生活帮”产品的长期唯一主参照，用于统一产品目标、业务范围、状态原则、权限边界与版本路线。
- 当其他文档与本文档冲突时，按用户最新明确决定与 `docs/product/PRODUCT_DECISION_LOG.md` 优先，其次再参考当前版本验收基线。
- 本文档不把当前代码行为直接等同于正确产品设计；代码仅作为现状输入之一。
- V3.2.4 的定位是“全量审查与验收基线统一版本”，不是新增大范围业务的扩展版本。

## 2. 产品定位

- 生活帮是一个围绕“生活需求撮合、专业服务履约、二手物品流转、回收与置换”的本地生活服务与交易平台。
- 平台同时服务普通用户、工程师、商家或回收方，以及平台审核与运营角色。
- 产品目标不是单一信息展示，而是把“发现问题、发起需求、获得响应、完成交易、保留记录、处理争议”串成可追溯的业务闭环。

## 3. 用户价值

- 对普通用户：用更低门槛发布需求、出售闲置、发起置换、询价回收，并获得消息、通知、订单与信用记录的持续承接。
- 对工程师：围绕专业服务场景查看需求、提交方案与报价、推进项目履约、交付文件并完成验收。
- 对商家或回收方：围绕回收与供给场景查看询价、提交报价、承接订单并管理过程状态。
- 对平台：通过认证、投诉、财务、审计与通知机制控制风险，并保留完整留痕。

## 4. 品牌级长期产品方向

- `生活妙招`：沉淀真实生活经验、方法和技巧。
- `生活认知`：总结可复用的生活规律、原则和判断框架。
- `奇思妙想`：承接想法验证、可能世界和新方向试验。
- `互帮互助`：承接当前需求、工程师、旧物、置换和回收平台的主要业务。
- `未来可期`：支持决策、执行、成长和长期个人发展。
- 上述方向是已确认的长期顶层产品方向，不直接计入 V3.2.4 未实现缺陷。

## 5. 用户角色

- 普通用户：浏览、搜索、发布需求、发布物品、发起回收、发起置换、下单、收发消息、接收通知。
- 工程师：在通过认证后查看需求、提交方案与报价、参与项目、上传文件、推进履约。
- 商家或回收方：在通过认证后查看回收请求、提交报价、承接回收类订单。
- 平台审核角色：处理认证、投诉、财务、审计与运营高风险动作。
- 未登录访客：仅可浏览公开内容，不可执行需要归属、支付、消息、文件或身份的操作。

## 6. 核心业务领域

- 需求撮合：需求、方案、报价、项目、里程碑、变更、验收。
- 交易履约：订单、支付、托管、退款、结算、投诉。
- 物品循环：物品档案、发布、一口价、置换、回收、赠送占位。
- 沟通通知：会话、消息、回执、站内通知、Push Token、实时送达。
- 账户治理：登录、资料、认证、信用、文件、安全、审计、后台权限。

## 7. 当前核心业务

- 当前产品已经规划或具备基础的核心业务能力：
  - 登录与登录恢复。
  - 首页、发现、搜索的公开浏览与分发。
  - 需求发布、需求详情、方案/报价、项目创建与履约。
  - 物品发布、编辑、图片上传、物品管理、一口价信息展示。
  - 置换、回收的核心流程。
  - 订单、消息、通知、我的、设置。
  - 认证、信用、投诉、财务与审计的后端能力基础。
- V3.2.4 是否必须通过，以 `docs/releases/V3_2_4_ACCEPTANCE_BASELINE.md` 为准。
- 当前版本可以使用沙箱或 Stub 方式完成外部服务适配，但不得声称已接入真实生产支付、真实生产 Push、真实对象存储或真实病毒扫描。

## 8. 后续扩展业务

- 后续可扩展到：
  - 赠送。
  - 捐赠。
  - 租借。
  - 拍卖。
  - 新品共创。
  - 更成熟的商家供给体系与运营后台。
  - 云端部署、云存储、正式 Push、金额迁移、真实支付与生产级风控。

## 10. 业务状态原则

- 所有高价值业务都必须通过后端状态机流转，不允许前端直接改写最终状态。
- 订单、支付、项目、投诉、认证、置换、回收都必须具备明确的起始态、中间态、终态与失败态。
- 同一实体进入终态后，后续动作必须受限，除非通过受控补偿路径处理。
- “草稿”“进行中”“待确认”“待支付”“已完成”“已取消”“已拒绝”“已退款”等语义必须保持一致，不允许同义词混用造成歧义。
- FUTURE 和当前版本未纳入必过项的能力，不得在验收时被直接登记为 V3.2.4 缺陷。

## 11. 数据一致性原则

- 关键业务变更必须由服务端事务保护，优先“先锁后改”。
- 支付、退款、通知、消息、置换、回收报价等链路必须优先考虑幂等。
- 资金、信用、审计、文件访问、状态历史等记录必须追加留痕，不可静默覆盖。
- 物理物品事实与发布行为分离：物品档案不等于某一次发布。
- 主业务事务与外部副作用解耦：Push、外部支付、外部存储失败不能直接破坏本地事实记录。

## 12. 权限原则

- 权限必须以后端为准，不能以前端隐藏按钮代替权限控制。
- 后台权限、业务准入权限、资源归属权限必须分层处理。
- 工程师与商家资格由认证状态控制，而不是简单依赖用户基础角色字段。
- 高风险操作必须要求显式确认，并保留审计记录。
- 未登录用户仅可访问公开内容，不可访问需要归属校验、消息、文件、认证或资金链路的页面与接口。

## 13. 文件和隐私原则

- 私有文件必须采用受控访问与短时签名，不允许公开裸链长期暴露。
- 文件访问、敏感资料读取、后台审核动作必须记录访问或审计日志。
- 日志、报告、测试证据中不得暴露密码、Token、Push Token、私人消息内容、数据库密钥或真实签名 URL。
- 没有真实外部服务账号时，可以保留本地存储、Stub 和开发 Provider，但必须标记为未接入生产。

## 14. 移动端体验原则

- Android Development Build 必须可运行，且能够连接本地后端与 MySQL 开发环境。
- 页面必须考虑安全区、键盘遮挡、Android 返回键、弱网、权限拒绝和长页面滚动。
- 列表与详情至少要有加载态、空态、错误态与重试入口。
- 真机验收结论必须与自动化证据分开记录，未真机验证的能力不得标记为“已通过”。
- 局域网地址只允许出现在被忽略的 `.env` 与 `.env.local` 中，不写入受 Git 跟踪源码或公共配置。

## 15. 非功能要求

- 保持当前技术栈：Expo 54、React Native 0.81、Expo Router、TypeScript、Express、tRPC、MySQL 8、Drizzle ORM、TanStack Query、NativeWind、pnpm。
- 迁移只允许追加，不允许修改已发布迁移。
- 每次交付至少应通过 `pnpm check`、`pnpm lint`、`pnpm test`、`pnpm build`、`pnpm build:web`。
- 涉及数据库、支付、投诉、物品、消息、文件或 WebSocket 时，还必须通过对应 MySQL 集成与迁移校验。
- `/api/health` 与 `/api/ready` 是运行基线；`ready` 失败必须明确反映配置或依赖未就绪。

## 16. 版本路线

- V3.2.1：稳定性与安全基线，重点是迁移安全、健康检查、文件安全、通知幂等、审计与规则收口。
- V3.2.2：核心产品闭环，重点是物品发布与管理、置换、回收、消息通知、页面四态与产品可见能力。
- V3.2.3：移动端准备，重点是 Expo SDK 54 兼容、Push 底座、权限处理、局域网诊断、可重复 Seed。
- V3.2.4：全量审查与验收统一版本，重点是全程序审查、Android 真机验收、问题冻结与体验收口，不新增大范围业务。
- NEXT_RELEASE：补齐云端能力、正式 Push、对象存储、移动端体验优化、更多运营工具。
- FUTURE：赠送、捐赠、租借、拍卖、新品共创等更大范围业务扩展。

## 17. 明确不属于当前阶段的事项

- 正式支付接通与生产对账。
- 金额体系迁移。
- S3 或同类云存储正式接入。
- ClamAV 或正式病毒扫描接入。
- 正式生产 Push Receipt 与稳定后台投递体系。
- iOS 真机验收作为 V3.2.4 必过项。
- 应用商店正式发布。
- 完整后台管理平台重构。
- 将 FUTURE 功能直接并入当前缺陷范围。
