# V3.3.0 第一阶段：金额迁移前置基线

## 1. 阶段目标

在不改写现有业务表、不批量乘以 100、不启用小数交易的前提下，建立 V3.3.0 金额迁移的唯一字段清单、统一分单位契约、只读数据库预检和专项测试，为后续影子列迁移与双写切换提供安全入口。

## 2. V3.2.4 当前判定

- 代码、规范与 Android 审查已完成收口。
- `validate:product-specs` 和 TypeScript 基线通过。
- V3.2.4 仍为 `RELEASE BLOCKED`，未完成稳定测试 API、最终 RC APK、21 项最终冒烟和正式发布门禁。
- V3.3 必须在独立本地目录继续，不能反向覆盖 V3.2.4 的冻结文档和历史验证证据。

## 3. 第一阶段已执行范围

### 3.1 统一金额契约

- 外部输入优先使用十进制字符串，例如 `"12.30"`。
- 领域计算统一转换为整数分 `bigint`。
- 禁止用浮点数静默四舍五入；`0.1 + 0.2` 之类的漂移输入会被拒绝。
- 兼容 V3.2 的整元 INT 调用，但明确标记为 legacy。
- 新增有符号金额支持，仅用于项目变更等允许负数的差额字段。

### 3.2 金额字段全量盘点

自动识别并登记 30 个金额字段：

- 17 个历史 `INT` 整元字段。
- 13 个 `DECIMAL(14,2)` 元账本字段。
- 覆盖工程师、需求、项目、旧物、回收、订单、物品、支付、退款、托管、结算和投诉资金动作。

任何后续新增金额字段，如果未进入注册表，`pnpm check:money:v330` 必须失败。

### 3.3 只读 MySQL 预检

新增 `pnpm audit:money:v330`，只读取数据库并检查：

- Schema 类型、可空性与注册表是否一致。
- 不允许负数的字段是否出现负值。
- 需求最低预算是否大于最高预算。
- 最低接受价是否高于挂牌价。
- 项目里程碑合计是否超过项目总额。
- 支付金额是否与订单金额一致。
- 成功退款合计是否超过支付金额。
- 托管入账、释放、退款是否越界。
- 必填金额是否出现异常零值。

脚本不创建表、不更新数据、不执行修复。

### 3.4 本地源码包适配

`check-markdown-links` 在没有 `.git` 的干净源码包中会自动扫描文件系统，不再强制依赖 Git 仓库。

## 4. 本阶段明确不做

- 不新增 Drizzle 迁移。
- 不修改任何现有金额列类型或含义。
- 不把历史整元值乘以 100。
- 不允许前端输入小数业务金额。
- 不接入正式支付 Provider。
- 不修改 V3.2.4 冻结产品规范。

## 5. 验收命令

```bash
pnpm validate:product-specs
pnpm check:markdown-links
pnpm check:money:v330
pnpm check
pnpm lint
pnpm test

```

```bash

```
